# YAHYA Sales Dashboard — заметки для Claude

## Что это
Streamlit-дашборд по продажам сети YAHYA (шоколад, кофейня, кухня).
Хостится на Streamlit Cloud, данные раздаются через GitHub Pages.

Репозиторий: `MonkeyDjack/yahya-sales-dashboard` (приватный).
Главный файл приложения: `dashboard_v2/app2.py`.

## Источники данных

| Файл | Что | Где раздаётся |
|---|---|---|
| `docs/база.parquet` | основная база продаж (~370 тыс строк, период 2025-01-01 → ...) | GitHub Pages |
| `docs/list1.parquet` | справочник SKU (Группа/Категория/Подкатегория) | GitHub Pages |

Дашборд кешит загрузку через `@st.cache_data(ttl=3600)` — изменения в parquet проявляются до 1 часа.

## Структура продаж (parquet)

Колонки: `Дата` (Timestamp), `Время` (str), `Филиал`, `Точки`, `Номенклатура`,
`Склад/Товар` (= чек), `Количество`, `Цена`, `Сумма`, `Месяц`, `Год`,
`Группа`, `Категория`, `Подкатегория`.

### Филиалы (на 2026-05)
- Старые (с 2025): ЦУМ, АЗИЯ МОЛЛ, ЭРКИНДИК, Бишкек Парк, АЭРОПОРТ
- Новые (открыты в 2026): NGROUP, АЗБУКА
- **АЗИЯ МОЛЛ** был на реконструкции **08.02.2026 – 01.03.2026** (22 дня) —
  это объясняет провал по февралю 2026 и **не** является сигналом падения спроса.

### Известные особенности данных
- SKU с суффиксом «**В**» в конце = «Вынос». Себес и розница такие же,
  как у базовой версии без «В». `build_cost_reference.py` обрабатывает это
  через расширенный `variants()`.
- В прайсах себеса часто бывают **«1 шт»** и **«(1конфета)»** в скобках,
  скобки нормализуются при матчинге.
- Покрытие себесом по выручке: **~84.5%** (Шоколад/Кухня/Бар).
  Без себеса остаются: сезонные (нет файла), шококарандаши (нет в прайсе),
  снятые с производства SKU.

### Бэкапы parquet
Перед массовыми правками `база.parquet` создаются `.bak_*`-файлы:
- `база.parquet.bak_before_category_fix` — до фикса категорий
- `база.parquet.bak_before_may2026`, `_before_fact16`, `_before_fact17` — до append'ов
Их **не коммитят** в репо (большие, не нужны).

## ABC-вкладка (tab 3)
Полностью переписана в `dashboard_v2/app2.py`. Поддерживает:
- **Режим «Один период»** + **Режим «Сравнение двух периодов»**
- Свой выбор периода (не глобальный date-фильтр)
- Пресеты: YTD vs прошлый год · Последние 30 vs пред. 30 · Месяц vs прошлый год · Квартал vs прошлый год
- Авто-обрезка длинного периода до короткого
- 4 уровня агрегации (Номенклатура / Подкатегория / Категория / Группа)
- 3 метрики (Сумма / Количество / Маржа)
- 6 под-вкладок при сравнении: Pareto-Drift · ABC-матрица переходов ·
  Growers · Decliners · Новинки/Снятые · Кандидаты на вывод

В коде использует `df_universe` (df с глобальными фильтрами, но БЕЗ date-фильтра)
и `cost_ref` (справочник себеса).

## Excel-форматирование
**Важно**: openpyxl по умолчанию НЕ пишет `applyFill="1"` в `<xf>`,
из-за чего старые версии Excel и LibreOffice **игнорируют заливку**
(но показывают цвет шрифта). Решение — использовать **xlsxwriter**
(`abc_chocolate_report.py`).

Если будешь писать Excel через openpyxl и нужны заливки — используй
форму `PatternFill(start_color="...", end_color="...", fill_type="solid")`,
не `PatternFill("solid", fgColor="...")`.

## Категории — частые проблемы
1. **Подкатегория = «Без подкатегории»** — после `fix_categories.py`
   автоматически заменяется на название Категории.
2. **SKU = «Конфеты / Корпусные конфеты»** как fallback — багованный импорт
   из 1С ставит это часть строк у произвольных SKU. `fix_categories.py`
   убирает через выбор канона по моде.
3. **Нац.коллекция × 6** — 5 из них канонично «Наборы и коробки / Наборы /
   Нац. конфеты», 1 («соленая карамель и талкан») в исходных данных была
   «Плитки / Нац. плитки» — есть ручной оверрайд в `fix_categories.py`.
4. **Шоколадные наборы из «Наборы и коробки»** (Нац.коллекция, Корпусная/Нарезная/
   Коктельная коллекция, Набор из дубайских мини плиток, Слива-Грецкий НК, Ынтымак)
   биз-логически — шоколад. В отчётах по шоколаду включаются явным списком префиксов.
